06 · SOLID 원칙

SOLID 원칙 적용

SRP

단일 책임 원칙

코드 위치: tokenizer.py · parser.py · checker.py · executor.py · loader.py · debugger.py

토큰화, AST 조립, 정적 검증·최적화, 실행, 모듈 로드, 디버깅을 분리합니다. 기능 하나를 바꿔도 영향과 테스트 범위를 해당 모듈로 좁힐 수 있습니다.

OCP

개방-폐쇄 원칙

코드 위치: ast_nodes.py의 Expr/Stmt.accept() · checker.py/executor.py의 visit_Xxx

AST는 방문자에게 디스패치하고, 분석·실행 행위는 visitor로 더합니다. 새 노드 처리 누락은 즉시 오류로 드러나 확장 과정의 조용한 누락을 막습니다.

LSP

리스코프 치환 원칙

코드 위치: runtime.py의 CodeFabFunction/CodeFabClass · executor.py의 visit_CallExpr()

두 타입은 CodeFabCallable의 arity()/call() 규약을 지켜 같은 호출 경로를 통과합니다. 함수 호출과 클래스 인스턴스화가 동일한 문법과 인자 검사를 공유합니다.

ISP

인터페이스 분리 원칙

코드 위치: runtime.py의 CodeFabCallable · executor.py의 visit_CallExpr()

호출부는 arity()와 call()만 의존합니다. bind()나 find_method() 같은 구현별 기능이 바뀌어도 일반 호출 로직은 영향을 받지 않습니다.

DIP

의존성 역전 원칙

코드 위치: executor.py의 Executor(..., loader, on_stmt) · factory_shell.py의 run_debug_mode()

Shell이 Loader와 DebugController.on_stmt를 조립해 실행기에 전달합니다. 디버그 정책을 바꿔도 실행 루프는 유지되며, 콜백 spy로 문장 단위 동작을 검증할 수 있습니다.

12 · Best Practice · 2/3

리뷰가 발견한 for 변수 스코프 누수

Code Review · Scope

dd2177e → f282c92 → 3807c7b · PR #20

INITIAL

현재 환경에서 그대로 실행

ForStmt의 초기화식·조건식·증감식을 self._current에서 처리해 var i = 0;이 전역 환경에 남았습니다.

self._exec_stmt(stmt.initializer)
while self._is_truthy(...):
    self._exec_stmt(stmt.body)
REVIEW

회귀 테스트 확장 중 발견

REPL 전역 스코프 인지 버그를 리뷰하며 테스트를 넓히다, 루프 뒤 var i = 5;가 중복 선언 오류가 되는 문제를 발견했습니다. PR의 의도는 맞았지만 별개의 누수 버그가 드러난 경우입니다.

for (var i = 0; i < 3; i = i + 1) { }
var i = 5;  // should be allowed
FIX

loop_env를 만들고 반드시 복원

루프 전용 환경으로 전환하고 finally에서 복원했습니다. 본문 오류가 나도 외부 스코프는 항상 정상 상태로 돌아옵니다.

loop_env = Environment(parent=self._current)
self._current = loop_env
try: ...
finally: self._current = prev

for 변수의 전역 누수를 막고, 단일·중첩 루프 누수와 같은 이름 섀도잉을 검증하는 회귀 테스트 3종을 추가했습니다.

13 · Best Practice · 3/3

디자인 패턴으로 구조를 리팩토링

Refactoring · Facade / Visitor

Facade PR #7 / #14 · Visitor PR #47

Facade

CodeFabInterpreter와 Assembler로 조립 순서 은닉

클라이언트는 run() 또는 assemble()만 호출하고, 하위 컴포넌트의 생성·연결 순서는 Facade가 맡습니다. Shell이 파이프라인 세부 순서를 알 필요가 없어 변경 범위가 줄었습니다.

Client→Facade→Assembler→Checker / Executor
Visitor

AST 변경 없이 새 순회 기능을 추가

Checker와 Executor가 accept(visitor)로 같은 AST를 다른 목적으로 순회합니다. 새 문법은 필요한 visit_Xxx를 구현하게 해 누락을 드러내고, 새 분석기는 AST 노드를 고치지 않고 추가할 수 있습니다.

Expr / Stmt→accept(visitor)→Checker / Executor

단순 구현을 넘어 변경을 한 곳으로 모으고 확장 시 누락을 빠르게 발견하는 구조로 발전시킨 리팩토링 사례입니다.

15 · 소감 1/2

소감

박준용

AI도구 사용으로 코드 생산성이 크게 향상된 만큼, 이를 효과적으로 통제하기 위한 클린 코드, 리팩토링, TDD의 중요성도 함께 커지고 있다는 것을 느꼈습니다. 그동안은 현업이 바쁘다는 이유로 테스트 코드 작성을 소홀히 하는 경우가 있었지만, 이제는 AI 도구를 활용해 테스트를 잘 작성해야겠다고 생각했습니다.

AI는 많은 양의 코드를 빠르게 생성해 주지만, 그만큼 요구사항을 정확히 검증할 수 있는 테스트 코드의 역할이 더욱 중요하다는 점을 다시 한번 깨달았습니다. 이번 프로젝트를 진행하면서도 초기에는 AI가 높은 수준의 코드를 생성해 주었지만, 프로젝트 규모가 커질수록 일부 요구사항을 놓치거나 의도와 다른 구현을 하는 경우가 있었습니다. 결국 AI가 생성한 코드도 코드리뷰 과정을 통해 꼼꼼하게 리뷰하고 검증하는 과정이 반드시 필요하며, 이를 뒷받침하는 테스트 코드와 코드 리뷰 문화가 앞으로 더욱 중요해질 것이라고 생각합니다.

권은재

TDD로 Checker를 만들다 보니 테스트를 먼저 정해두는 게 이후 기능이 추가될 때도 회귀를 잡아주는 안전장치가 된다는 걸 실감했습니다. 다른 기능이 merge될 때마다 제가 만든 로직과 충돌하는 부분이 없는지 직접 코드를 추적해서 확인해야 했고, 그 과정에서 리뷰는 통과 여부만 보는 게 아니라 왜 그렇게 동작하는지까지 확인하는 작업이라는 걸 다시 배웠습니다. 리팩토링도 몇 차례 겪었는데, 구조가 바뀌어도 테스트가 그대로 통과하는지로 검증할 수 있어서 수업 목표였던 TDD·리팩토링을 실제로 써볼 수 있었던 프로젝트였습니다.

김종화

이번 프로젝트를 통해서 처음부터 끝까지 TDD를 적용하여 개발하는 경험을 처음 해보았는데, 모든 변경점이 UT를 통해 정확히 동작하는것이 검증되고 있어 안심하고 개발할 수 있었습니다. 기존에 제품 개발할 때에는 이런 안전장치가 없어서 늘 불안한 마음을 가지고 있는데, 앞으로 현업에 돌아가서 UT 작성을 꾸준히 해야겠다고 생각했습니다.

16 · 소감 2/2

소감

송지영

현업에서는 주로 혼자 개발하다보니 클린코드나 TDD의 필요성을 크게 느끼지 못했는데, 팀 프로젝트를 해보니 팀원의 코드를 이해하거나 수정할 때 도움이 되었습니다. 혼자 개발하더라도 언젠간 다른 사람이 내 코드를 보거나 수정해야하는 상황이 올 수 있다는 것을 고려하여 test 코드를 추가하고 리팩토링 잘 해둬야겠다는 생각이 들었습니다.

이채연

CLAUDE를 사용하여 처음부터 끝까지 개발해본 프로젝트인데, 팀원이 모두 다 같은 설계 문서만 공유한다면 CLAUDE를 사용하는 것이 코드 스타일이나 설계 관점에서 협업에 더 유리하다고 느꼈습니다. 모듈을 나눠서 각자 독립적으로 개발해도, 팀원 모두가 테스트 케이스를 꼼꼼하게 정의해둔 덕분에 다른 모듈들이 제 모듈 변경사항으로 인해 깨지진 않는지 쉽게 검증할 수 있어서 개발이 더 수월했다는 걸 깨달아서, 현업에 가서도 TC를 보강해야겠다고 느꼈습니다.

조재현
