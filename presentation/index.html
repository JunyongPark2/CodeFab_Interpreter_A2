02 · 기능 구현

기능 구현 소개 (구조 설계)

사용자 입력은 CodeFabInterpreter 퍼사드를 거쳐 Assembler → Checker → Executor 순서로 처리됩니다. AST는 한 번 만들고, 정적 검사와 실행 단계에서 Visitor 방식으로 재사용합니다.

Assembler

소스를 토큰화하고 AST 문장 목록으로 조립합니다.

02

03 · SOLID & 디자인 패턴

SOLID 원칙과 적용 패턴

CodeFab Interpreter의 설계는 요구사항이 늘어날 때 변경 범위를 줄이는 방향으로 정리되어 있습니다. 발표에서는 실제 코드 지점과 함께 세 가지 패턴만 집중해서 설명합니다.

SRP

단일 책임 원칙

Tokenizer, Parser, Checker, Executor, Loader, DebugController가 단계별 책임을 나눕니다.

OCP

개방-폐쇄 원칙

AST 노드가 늘어나면 visit_Xxx 메서드를 추가하는 방식으로 확장하고, 누락 시 accept()가 즉시 실패합니다.

LSP

리스코프 치환 원칙

CodeFabFunction과 CodeFabClass가 모두 CodeFabCallable로 호출되어 같은 실행 경로를 공유합니다.

ISP

인터페이스 분리 원칙

호출 가능한 값은 arity()/call()만 요구하고, 디버거는 on_stmt 훅만 제공합니다.

DIP

의존성 역전 원칙

Assembler와 Executor는 tokenizer/parser/loader/hook을 주입받아 테스트 더블로 교체하기 쉽습니다.

Facade

CodeFabInterpreter.run()

사용자는 run() 하나만 호출하고, 내부 파이프라인 조립은 퍼사드가 맡습니다.

Visitor

Expr/Stmt.accept()

같은 AST를 Checker와 Executor가 각각 다른 목적으로 순회합니다.

Strategy / Callback Hook

Executor(on_stmt=...)

일반 실행과 디버그 실행의 차이를 문장 실행 전 콜백 주입으로 분리합니다.

  • import는 Loader와 Executor의 책임을 나누어 기존 함수/클래스 실행 로직과 충돌을 줄였습니다.
  • debug next는 Executor가 depth와 hook만 제공하고, 정지 판단은 DebugController가 맡도록 개선했습니다.
  • 상수 폴딩과 정적 바인딩은 Checker에서 계산하고 Executor가 결과를 사용해 실행 단계의 부담을 줄였습니다.

09 · Best Practice

BP 사례

실제 PR/커밋에서 확인한 Code Review, TDD, Refactoring, 사용성 개선 열 가지 사례입니다.

Code Review

클래스명 중복 선언 검출 누락

- Class 선언 시 스코프 등록 없음+ visit_ClassDeclStmt()에서 이름 선언 등록

TDD

modulo 상수 폴딩 누락

PDF 예시의 % 리터럴 연산이 매 반복마다 재계산되는 문제를, 실패하는 회귀 테스트를 먼저 작성해 고정한 뒤 Checker._try_fold_binary()를 수정했습니다.

Refactoring

eval_binary_op 추출

Checker의 상수 폴딩과 Executor의 실행 로직에 중복된 11개 이항 연산자 처리를, runtime.eval_binary_op 한 곳으로 통합했습니다. 이후 연산 의미가 바뀌어도 한 곳만 수정하면 됩니다.

- checker.py, executor.py 각각 구현+ runtime.eval_binary_op() 단일 소스

파일 실행 · 디버그 · REPL 세 경로 모두에서 ModuleImportError가 사용자에게 traceback으로 새던 문제를 찾아, 세 진입점 모두를 메시지 출력 + 종료 코드 1로 통일했습니다.

10 · 시연 로드맵

기본 기능과 엣지 케이스 시연 목록

Basic Flow

언어 기능이 한 파이프라인에서 끝까지 실행되는지 확인

  1. REPL / 멀티라인 입력변수, print, if 블록이 같은 전역 환경을 공유
  2. Function인자, return, 재귀, null 반환
  3. ClassThis, Super, init, 상속, instanceof
  4. Array / Modulo고정 크기 배열, 인덱스 읽기/쓰기, % 연산
  5. Import모듈 네임스페이스와 alias 멤버 접근
  6. Debug Shellstep, next, watch, inspect가 Executor hook으로 동작
Edge Cases

리뷰와 회귀 테스트에서 잡은 실패 경로를 짧게 재현

  1. 정적 검사클래스명/alias 중복, 함수 밖 return, init return 금지
  2. 배열 런타임 오류음수·소수 크기, 범위 밖·비정수 인덱스
  3. Import 오류 UX누락 파일, 순환 import, 재import, for 내부 import 금지
  4. 상수 폴딩 안전장치% 0과 타입 오류는 실행 전 접지 않음
  5. Debug next복합문 내부에서 다시 멈추지 않는지 확인
  6. 스코프 경계섀도잉 허용, for 변수 누수 방지, REPL import 기록 유지

12 · 엣지 케이스 시연

실패 경로와 회귀 방지 데모

Checker

클래스명이 기존 변수명을 덮어쓰지 못함

python factory_shell.py run scripts/class_name_collision_repro.txt
[7번째줄] 변수 'Robot'이(가) 이미 이 스코프에 선언되어 있습니다.
Runtime

배열 크기와 인덱스 검증

var arr = Array(2);
arr[2] = 9;
// [2번째줄] 배열 인덱스가 범위를 벗어났습니다.

추가 후보: Array(2.5), arr[1.5], "x"[0]

Import UX

누락 파일은 traceback 없이 한 줄 오류

python factory_shell.py run scripts/missing_import_error_demo.txt
[1번째줄] import 대상 파일이 없습니다: 'missing_file_for_manual_check.txt'

추가 후보: 같은 파일 재import, 순환 import, for 본문 내부 import 금지

Optimizer

상수 폴딩은 런타임 오류를 숨기지 않음

print 10 % 3;
print 10 % 0;
// 1
// [2번째줄] 0으로 나눈 오류

runtime.eval_binary_op() 공유 이후에도 % 0은 접지 않고 Executor에서 오류가 납니다.

Static Scope

문맥이 맞지 않는 키워드는 실행 전 차단

return 1;                 // 함수 밖 return
Class A { init() { return; } }
print This;               // 클래스 밖 This
Class A { m() { Super.m(); } } // 부모 없는 Super
Debug UX

step과 next의 의미 차이를 직접 비교

scripts/debug_next_for_repro.txt
step  → for 내부 문장까지 진입
next  → 같은 깊이의 다음 문장까지 진행

발표 시간이 부족하면 기본 기능 시연의 Debug 항목 하나로 합쳐도 됩니다.
